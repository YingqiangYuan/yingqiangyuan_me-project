## Project at a glance

`yingqiangyuan_me` is YingQiang Yuan's personal portfolio website at https://yingqiangyuan.me. It serves two visitor experiences from a single repo: a marketing landing page (`/`) and a topic-locked AI chat (`/chat`) that answers only career questions about YingQiang. The frontend is Next.js 16 (App Router) deployed on Vercel; the backend is a FastAPI app at `api/index.py` deployed as Vercel Serverless Functions; the AI brain is AWS Bedrock (`us.amazon.nova-micro-v1:0`) called via the boto3 Converse API with prompt caching. The whole architecture is intentionally split: visual polish + DX in TypeScript, AI plumbing + AWS in a clean Python package (`yingqiangyuan_me/`).

## Architecture overview

```
Browser ── (Next.js dev :3000 / Vercel edge)
   │
   │   POST /api/chat   ── AI SDK v5 Data Stream Protocol (SSE)
   │                     + X-Client-Fingerprint header
   ▼
FastAPI ── api/index.py  (Vercel Serverless Function in prod, uvicorn :8000 in dev)
   │
   │   ai_sdk_adapter.py        : SDK v5 → Bedrock Converse format conversion
   │   one (singleton)          : ConfigMixin + Boto3Mixin → cached bedrock_runtime_client
   │   ChatSession              : multi-turn history wrapper around Converse API
   │   prompts/instruction.md   : system prompt (topic guardrails + jailbreak)
   │   prompts/knowledge-base.md: cached "first turn" with résumé context
   ▼
AWS Bedrock Runtime  (us.amazon.nova-micro-v1:0 — cross-region inference profile)
```

Two design choices that propagate everywhere:

1. **Singleton + Mixin composition.** `yingqiangyuan_me.one.api.one` is the sole entrypoint to all AWS state. Every backend handler does `from yingqiangyuan_me import one` and then `one.bedrock_runtime_client`. New cross-cutting concerns (DynamoDB, secrets, agents) become new mixins on `One`, never new globals.
2. **Local vs Vercel runtime detection.** `runtime.py` reads `VERCEL=1`. `Config` then has two factory paths (`new_in_local_runtime` calls `load_dotenv()` first; `new_in_vercel_runtime` reads env directly). Anything that needs to behave differently between dev and prod should look at `runtime` and stay in `config/`.

### Backend — entry & wiring

**`api/index.py`** *(160 lines)*
- **What.** FastAPI app exposing `GET /api/hello` (health) and `POST /api/chat` (the AI streaming endpoint).
- **Why.** Vercel mounts the file as a Serverless Function. FastAPI was chosen because the Python AI/AWS code already lives here — Next.js Route Handlers would have forced the AI logic into TS or required a separate service.
- **Key files.** `api/index.py`.
- **Depends on.** `fastapi`, `vercel_ai_sdk_mate`, `yingqiangyuan_me` (the singleton + adapter + ChatSession + paths), `rich` (debug pretty-printing).
- **Gotchas.**
  - The header it streams back is `x-vercel-ai-ui-message-stream`, *not* `x-vercel-ai-data-stream` — that's the AI SDK v5 spec; v4 used the latter.
  - Message length validation (lines ~78–91) is **commented out**. Re-enabling it is a one-uncomment fix.
  - `knowledge-base.md` is seeded as the **first conversation turn** plus a `cachePoint` so Bedrock caches it across turns. That's why ChatSession's history is non-empty before the user even types.

**`yingqiangyuan_me/__init__.py`** + **`yingqiangyuan_me/api.py`**
- **What.** Re-export the `one` singleton at three levels: `yingqiangyuan_me.one` (package), `yingqiangyuan_me.api.one` (public API surface), `yingqiangyuan_me.one.api.one` (internal).
- **Why.** Lets handlers say `from yingqiangyuan_me import one` while internal modules use the explicit path. Keeps the import surface stable as internals move.
- **Key files.** `yingqiangyuan_me/__init__.py`, `yingqiangyuan_me/api.py`, `yingqiangyuan_me/one/api.py`, `yingqiangyuan_me/one/__init__.py`.
- **Depends on.** `yingqiangyuan_me.one.api`.
- **Gotchas.** Three layers of re-export look redundant; the WHY is import-stability. Don't collapse them without thinking through what callers will break.

### Backend — runtime & config

**`yingqiangyuan_me/runtime.py`**
- **What.** Singleton `runtime` exposing `is_local()`, `is_vercel()`, `name`. Reads `os.environ.get("VERCEL", "NOTHING") == "1"`.
- **Why.** Centralises runtime detection so `Config` (and only `Config`) branches on it.
- **Key files.** `yingqiangyuan_me/runtime.py`.
- **Depends on.** stdlib only.
- **Gotchas.** Anything other than the literal string `"1"` is treated as local — including unset, `"0"`, `"true"`. Vercel sets `VERCEL=1` by convention; don't trust other CI providers' env var names.

**`yingqiangyuan_me/config/conf_00_def.py`**
- **What.** `Config` dataclass with `aws_region`, `aws_access_key_id`, `aws_secret_access_key`, `model_id` (defaults to `us.amazon.nova-micro-v1:0`). Class methods: `new()` (auto-route by runtime), `new_in_local_runtime()` (calls `load_dotenv()` first), `new_in_vercel_runtime()` (env-only).
- **Why.** All env-aware credential logic in one file. The `0_def` numeric prefix signals "this is the definition file; later files in `config/` extend it".
- **Key files.** `yingqiangyuan_me/config/conf_00_def.py`, `yingqiangyuan_me/config/__init__.py`.
- **Depends on.** `python-dotenv`, `runtime`.
- **Gotchas.**
  - Locally **and** on Vercel the body of credential loading is the same `_load_env_var()` helper — the only difference is the `load_dotenv()` call. Don't add divergent local-only logic without thinking about parity.
  - `model_id` is a Bedrock cross-region inference profile (`us.amazon...`), not a plain model id. Switching to a non-profile id is a breaking change in some regions.
  - **WHY (best guess; verify with project owner):** the numeric file prefix (`conf_00_def`, `one_00_main`, `one_01_config`, `one_02_boto3`) is "alphabetical = read order" — so a future reader naturally hits the most foundational file first.

**`yingqiangyuan_me/one/one_00_main.py`** + **`one_01_config.py`** + **`one_02_boto3.py`**
- **What.** `One` class composed of `ConfigMixin` (gives `self.config: Config`, lazy via `cached_property`) and `Boto3Mixin` (gives `self.bsm`, `self.boto_ses`, `self.bedrock_runtime_client`, all lazy).
- **Why.** Mixin composition keeps each concern in its own file (~20 lines each) and makes adding a new concern (say, a DynamoDB client) a 1-file change. Cached properties mean clients are only constructed when first used — meaningful for cold-start latency on Vercel.
- **Key files.** `yingqiangyuan_me/one/one_00_main.py`, `one_01_config.py`, `one_02_boto3.py`, `__init__.py`, `api.py`.
- **Depends on.** `boto3`, `boto-session-manager`.
- **Gotchas.**
  - `Boto3Mixin.bsm` is a `BotoSesManager` (used by `pynamodb-session-manager`); `bedrock_runtime_client` is what `api/index.py` actually calls. **PynamoDB is declared but unused today** (see "External dependencies" + 03-elevation-roadmap.md).
  - `cached_property` means the same `bedrock_runtime_client` instance is reused across requests in the same Lambda/Serverless container. Boto3 clients are thread-safe; this is fine.

### Backend — chat plumbing

**`yingqiangyuan_me/multi_round_bedrock_runtime_chat_manager.py`**
- **What.** `ChatSession` dataclass: holds `client`, `model_id`, `system` text, mutable `_messages` history, `_session_id`. `send_message(messages)` appends new turns, calls `client.converse(...)`, appends the assistant reply, returns the typed response.
- **Why.** Hides the verbosity of the Bedrock Converse API and gives a typed return value (via `boto3-dataclass`). Lets `api/index.py` stay short.
- **Key files.** `yingqiangyuan_me/multi_round_bedrock_runtime_chat_manager.py`.
- **Depends on.** `boto3`, `boto3-dataclass[bedrock-runtime]`, `func-args` (for `OPT` / `remove_optional()`), `rich` (debug printing).
- **Gotchas.**
  - `_messages` is **mutated in place** by `send_message`. Sharing one ChatSession across requests would leak state. The current code path constructs a new ChatSession per request inside the handler — keep it that way unless you add per-user storage.
  - `boto3-dataclass` casts the raw Bedrock dict response into a typed dataclass, so IDE autocomplete works on `response.output.message.content[0].text`. If you swap models, recheck the response shape.
  - `remove_optional()` strips `OPT`-marked fields before calling Bedrock. Useful when you want to omit `inferenceConfig` rather than send `None`.

**`yingqiangyuan_me/ai_sdk_adapter.py`**
- **What.** Format conversion + SSE generator. Three roles:
  1. Parse incoming Vercel AI SDK v5 request → list of Bedrock `messages`.
  2. Extract `last_user_message_text` for validation/logging.
  3. Generate the SSE response stream that the frontend's `useChat` hook expects (`text-start` → repeated `text-delta` → `text-end` → `finish` → `[DONE]`).
- **Why.** AI SDK v5 and Bedrock have different message shapes (`parts[]` of `{type: "text", text: ...}` vs `content[]` of `{text: ...}`); without an adapter, every handler would re-implement the conversion.
- **Key files.** `yingqiangyuan_me/ai_sdk_adapter.py`.
- **Depends on.** `vercel-ai-sdk-mate` (typed dataclasses for the SDK protocol), `fastapi.Request`, stdlib `json`/`uuid`.
- **Gotchas.**
  - **Text parts only.** Image/file parts in the request will be silently dropped (or worse). Adding multimodal requires extending `part_to_bedrock_content()`.
  - System messages from the SDK are skipped on conversion — Bedrock takes `system` as a separate top-level argument, not as a message.
  - The SSE event format must match what `@ai-sdk/react` parses; if you change event names, the frontend will appear to "do nothing" with no obvious error.

**`yingqiangyuan_me/paths.py`**
- **What.** `path_enum` singleton: filesystem path constants + `cached_property` accessors `instruction_content` and `knowledge_base_content` that read once and reuse.
- **Why.** Cold-start latency on Vercel is sensitive to filesystem reads; caching the prompt files in module memory makes the second request fast.
- **Key files.** `yingqiangyuan_me/paths.py`.
- **Depends on.** `pathlib`, `functools.cached_property`.
- **Gotchas.** Cached → **edits to `prompts/*.md` require a server restart** (or a new Vercel deploy) to take effect.

**`yingqiangyuan_me/utils.py`**
- **What.** `debug(s: str)` prints to `sys.stderr`. That's it.
- **Why.** Logs go to Vercel function logs (stderr) without polluting the response stream (stdout).
- **Gotchas.** No log level, no formatter. Adequate for a portfolio site; not a substitute for real observability (see 03-elevation-roadmap.md).

**`yingqiangyuan_me/prompts/instruction.md`** *(106 lines)*
- **What.** The system prompt. Defines two rules (only YingQiang, only knowledge-base facts), a refusal template, a decision tree, and explicit jailbreak protection.
- **Why.** This is the *product* of the chatbot: the topic constraint is the differentiator vs a generic ChatGPT clone. The jailbreak block exists because public chatbots get prompt-injected within hours of launch.
- **Gotchas.**
  - The personal contact details (email, phone, location) are hard-coded in the prompt itself, not just the knowledge base. Keep them in sync if YingQiang moves.
  - The refusal template is a fixed string — a clever attacker can detect "I only discuss YingQiang Yuan's professional background" verbatim. If that becomes a problem, paraphrase variants in the prompt.

**`yingqiangyuan_me/prompts/knowledge-base.md`**
- **What.** Structured résumé content (summary, education, skills, experiences). Injected as the **first user turn** in every conversation, with a `cachePoint` so Bedrock caches it.
- **Why.** Bedrock prompt caching only works for prefix tokens. By making the knowledge base the first turn, every subsequent message in the same session benefits from cache hits — visible cost reduction.
- **Gotchas.**
  - This file is generated/curated from `resume/2026-05-06-all-in-one-resume.md`. The two are not auto-synced — when YingQiang updates his résumé, this file must be regenerated.
  - The structure (markdown headings) matters. The model uses heading hierarchy to retrieve facts; collapsing into prose hurts answer quality.

### Frontend — entry & layout

**`app/layout.tsx`**
- **What.** Root layout. Loads Google fonts (Unbounded display, Manrope body, JetBrains Mono code), forces dark mode (`<html className="dark">`), wires `generateSEOMetadata()`.
- **Why.** Three-font setup is intentional — Unbounded for headlines = "designy", Manrope for body = readable, mono for code blocks in chat. Dark-mode-first matches the brand.
- **Gotchas.** `next-themes` is in `package.json` but the root `<html>` is hard-coded `dark`. If you ever ship a theme toggle, the wrapper needs to move into a client component.

**`app/(marketing)/`** — landing page route group
- **What.** `(marketing)` is a Next.js route *group* (parens hide it from URL). Contains `page.tsx` (renders `HomePageContent`), `layout.tsx` (wraps `CircuitBackground` + `Navigation`), `loading.tsx`, and `_components/{Hero,StatsSection,ContactSection}.tsx`.
- **Why.** The route group lets marketing and chat have totally different layouts (chat doesn't want the animated circuit background) without inventing a `/marketing` URL segment.
- **Gotchas.** The `_components/` underscore prefix is Next.js convention for "private to this route, do not auto-route". Don't move shared things in here.

**`app/chat/`** — chat route
- **What.** `page.tsx` renders the `Chat` component from `components/chat/chat.tsx`. `layout.tsx` adds the shared `Navigation` + top padding. `loading.tsx` is the skeleton.
- **Why.** Chat layout is deliberately minimal — no animated background — so the chat focus isn't fighting the page chrome.

**`app/_components/layouts/Navigation.tsx`** + **`app/_components/common/CircuitBackground.tsx`**
- **What.** Shared header (logo, nav links) and the SVG-animated circuit background used on marketing.
- **Why.** Lifted to `app/_components/` because they're used across multiple route groups.

### Frontend — chat UI

**`components/chat/chat.tsx`**
- **What.** The `Chat` component. Wires `useChat()` from `@ai-sdk/react`, manages input state, renders message list, handles streaming/stop/error/toast.
- **Why.** `useChat` does all the SSE parsing and message buffering — the component is mostly state glue and rendering.
- **Depends on.** `@ai-sdk/react`, `@fingerprintjs/fingerprintjs`, `sonner` (toasts), `framer-motion`, `hooks/use-scroll-to-bottom`.
- **Gotchas.**
  - `chatId` is hard-coded to `"001"`. There is no per-user / per-session storage. Multi-session is an explicit elevation candidate (see 03).
  - The fingerprint is injected by **wrapping `window.fetch`** so every request automatically gets `X-Client-Fingerprint`. If something else also wraps fetch, order matters.
  - Rate-limit error detection is string-matched on `"429"`, `"Too many requests"`, `"Rate limit"`. The backend currently doesn't 429, so this is dead code today — but it's wired and ready.

**`components/chat/message.tsx`** + **`markdown.tsx`** + **`overview.tsx`** + **`multimodal-input.tsx`**
- **What.** Message bubble (`PreviewMessage`) and "thinking" placeholder; markdown renderer with `react-markdown` + `react-syntax-highlighter`; welcome screen shown when `messages.length === 0`; the textarea + suggested-action buttons.
- **Why.** Splitting these out keeps `chat.tsx` short and lets each piece evolve independently (e.g. swapping the markdown renderer doesn't touch the chat state machine).
- **Gotchas.**
  - `multimodal-input.tsx` is named "multimodal" but only handles text today — naming is aspirational.
  - The textarea persists user input via `useLocalStorage("input")` — refreshing the page restores what the user was typing.

**`components/ui/*`** *(50+ files)*
- **What.** shadcn/ui — Radix UI primitives styled with Tailwind. Button, input, card, dialog, tabs, sonner toast, etc.
- **Why.** Industry-standard headless components; keeps a consistent design system without committing to a heavyweight library.
- **Gotchas.** Each file was generated by `npx shadcn-ui add <component>`. Customisations live inline in the file; don't expect upstream updates to merge cleanly.

### Tooling & config

**`mise.toml`** *(64 lines)*
- **What.** Tool versions (Python 3.12, Node 24, uv latest, pnpm 10.33.3) + tasks (`venv-create`, `inst`, `dev`, `kill`, `test-python`, `test-node`, `export`, etc.).
- **Why.** `mise` is the polyglot replacement for `nvm`+`pyenv`+`make` — one file pins everything, including Claude Code itself (`claude = "latest"`). The `dev` task `depends = ["kill"]` so stale servers can never collide with a fresh start.

**`pyproject.toml`**
- **What.** Python deps: `fastapi`, `uvicorn`, `boto3`, `boto3-dataclass[bedrock-runtime]`, `boto-session-manager`, `pynamodb`, `pynamodb-session-manager`, `vercel-ai-sdk-mate`, `pydantic`, `func-args`, `rich`, `python-dotenv`. Test extra: `pytest`.
- **Gotcha.** `pynamodb` and `pynamodb-session-manager` are declared but **not imported** by any file in `yingqiangyuan_me/`. They're scaffolding for a future feature (likely chat persistence; see 03).

**`package.json`** + **`package.json.md`**
- **What.** The Node dependencies: `next`, `react`, `ai` + `@ai-sdk/react`, `@fingerprintjs/fingerprintjs`, ~30 `@radix-ui/*` components, `tailwindcss`, `framer-motion`, `react-markdown`, `react-syntax-highlighter`, `recharts`, `sonner`, `next-themes`, `concurrently`, `usehooks-ts`, etc. The `.md` sibling explains the intent of each dep — JSON has no comments.
- **Gotcha.** `package.json.md` is a **legitimate doc**, not scratch. Keep it in demo viewability.

**`next.config.js`**
- **What.** Image `remotePatterns` for `*.yingqiangyuan.me` CDNs; rewrites `/api/*` → `http://127.0.0.1:8000/api/*` in dev, `/api/` direct in prod.
- **Why.** The dev rewrite is what lets the frontend at `:3000` talk to FastAPI at `:8000` without CORS. In prod, Vercel routes `/api/*` to the Serverless Functions natively.

**`vercel.json`**
- **What.** `functions["api/**"]` config; build env `PYTHON_VERSION=3.12`, `ENABLE_EXPERIMENTAL_COREPACK=1`.
- **Why.** Tells Vercel which Python to use and to enable corepack (so `pnpm` works in Vercel's build).

**`tailwind.config.ts`**, **`tsconfig.json`**, **`postcss.config.mjs`**, **`next-env.d.ts`** — standard tooling configs; nothing surprising.

**`kill-dev-servers.py`** *(273 lines, executable)*
- **What.** Parses `ps aux`, identifies Next.js processes (keywords: `next-server`, `next dev`, `concurrently`, etc.) and FastAPI processes (`uvicorn`, `api.index:app`, etc.), kills them by PID with `kill -9`. Falls back to `lsof -ti :3000` etc. if PID detection misses.
- **Why.** `mise run dev` runs Next.js + FastAPI under `concurrently`. If the parent dies abruptly (Ctrl-Z, SIGKILL on the terminal, IDE crash), the children become orphans bound to ports 3000/3001/3002/8000/8080. This script is the recovery path. `mise run dev` calls `mise run kill` first as a depends, so the failure mode is invisible most of the time.
- **Gotchas.** `kill -9` is ungraceful. If you ever add background workers that need flush-on-shutdown, this script will eat their data. The graceful path is `kill -15` first, wait, then `-9`.

## Conventions

- **Numeric file prefixes (`conf_00_def`, `one_00_main`, `one_01_config`).** Read-order convention. Lower number = more foundational. Insert new files between with `0X_5_...` if needed.
- **Mixin composition for `One`.** Each cross-cutting concern is a `*Mixin` in `yingqiangyuan_me/one/` that contributes one or two `cached_property` accessors.
- **Singletons via module-level instance.** `runtime` (in `runtime.py`), `path_enum` (in `paths.py`), `one` (in `one/api.py`). Don't instantiate the classes elsewhere.
- **Cached properties for lazy AWS clients.** Cold-start matters; pay the boto3 init cost on first use, not on import.
- **Public API re-exports at three levels.** `yingqiangyuan_me`, `yingqiangyuan_me.api`, `yingqiangyuan_me.one.api` — keep the public surface stable while internals move.
- **`(marketing)` route group + `_components/` private folder.** Next.js conventions, used deliberately.
- **`cn()` instead of conditional class strings.** All Tailwind class composition goes through `lib/utils.ts:cn`.
- **Static JSON / TS for "data" files.** No CMS. `data/achievement-stats.ts`, `data/suggested-actions.json`.
- **Header-driven SSE event names.** Backend emits the literal AI SDK v5 event names (`text-start`, `text-delta`, `text-end`, `finish`, `[DONE]`). Don't paraphrase.
