# 06 — Demo Playbook

> How to demo `yingqiangyuan_me` effectively, tailored by audience. Honesty first: if a feature is half-done, the script says so — never paper over it. The "Do NOT show" list is the most-load-bearing part of this doc.

## Wow features (in priority order)

### 1. The topic-locked AI chat with jailbreak resistance

**What it is.** The chat at `/chat` answers questions about YingQiang's career — and only those. Try to ask it for a translation, a math problem, code help, or general career advice and it politely refuses with a fixed template, then offers to talk about YingQiang instead.

**Why it lands.** It's not the LLM that's impressive — it's the *prompt engineering discipline*. The system prompt has explicit topic boundaries, a refusal template, a decision tree, and jailbreak defenses against "ignore previous instructions" attacks. Anyone who's tried to ship a chatbot knows how hard topic-locking actually is. Demoing the *refusal* (then a successful answer about YingQiang) shows control.

**Where to point.** `/chat` page. After demonstrating the topic-lock, open `yingqiangyuan_me/prompts/instruction.md` to show the rules.

### 2. Real-time streaming chat with prompt caching

**What it is.** The chat replies token-by-token via SSE, not in a single block. Behind the scenes, the knowledge base is injected as the first conversation turn with a Bedrock `cachePoint`, so subsequent turns hit the cache and pay near-zero input-token cost.

**Why it lands.** The streaming alone is table-stakes. The caching story is what separates a "called an API" project from "thought about cost". If your interviewer cares about ML systems, this is the beat to land on.

**Where to point.** `/chat` again — let the user watch the streaming. Then open `api/index.py` and show the cachePoint placement.

### 3. Animated landing page with intentional design system

**What it is.** The landing page at `/` opens with an animated SVG circuit-board background (`CircuitBackground.tsx`), a hero section, an animated stats grid (Recharts), and a contact section. Three Google fonts (Unbounded for display, Manrope for body, JetBrains Mono for code) and a strict dark-mode design system.

**Why it lands.** Visual polish signals that the engineer cares about end-to-end UX, not just backend. For a portfolio, this is non-negotiable. For other audiences, it's secondary.

**Where to point.** `/` landing page. Optionally show `tailwind.config.ts` for design tokens or `app/_components/common/CircuitBackground.tsx` for the SVG animation.

## Recommended sequence — the golden path

### 5-minute version (focused; default for most audiences)

1. **(0:00–0:30) Open** http://yingqiangyuan.me (or the local `mise run dev` instance at `:3000`). Land on the marketing page. **Say:** "This is YingQiang Yuan's portfolio. The marketing page is Next.js — what I want to show you is the chat."
2. **(0:30–1:30) Click "Chat"** in the navigation. Land on `/chat`. **Say:** "This is an AI chatbot that answers career questions about YingQiang. The interesting part is what it *won't* do." Type: *"Translate this to French: hello world"*. **Watch the polite refusal stream.** **Say:** "It's topic-locked — that's a system prompt with explicit jailbreak defenses, not a model finetune."
3. **(1:30–3:00) Now ask a real question.** Type: *"What did YingQiang do at EasyScaleCloud?"* **Watch the streaming response.** **Say:** "Streaming is via Server-Sent Events using Vercel's AI SDK v5 protocol. The response is grounded in a cached knowledge base — I inject the résumé as the first conversation turn with a Bedrock `cachePoint`, so the second turn onwards pays near-zero input-token cost."
4. **(3:00–4:00) Open the code briefly.** Show `yingqiangyuan_me/prompts/instruction.md` (the system prompt) and `api/index.py` (where the cachePoint is set). **Say:** "Two files: rules and facts. Separation of concerns."
5. **(4:00–5:00) Wrap.** **Say:** "Stack-wise: Next.js 16 frontend, FastAPI backend on Vercel Serverless, AWS Bedrock Nova Micro with prompt caching. The Python package uses a singleton + Mixin pattern so adding a new AWS service is a one-file change. Things I'd ship next: server-side rate limiting and structured observability — both planned, both on the roadmap."

### 15-minute version (deep dive; for engineers / technical hiring managers)

1. **(0:00–1:00) Project framing.** Open the landing page. Pitch the project in 90 seconds (use the model pitch from `05-interview-playbook.md`).
2. **(1:00–3:00) Architecture diagram.** Open `docs/learn-this-project/01-knowhow-inventory.md#architecture-overview` and walk the diagram. Name each hop: Browser → Next → FastAPI → Bedrock. **Say:** "Two runtimes — Next for UI, Python for AI plumbing — with a clean rewrite at the boundary."
3. **(3:00–5:00) Chat demo: the refusal first.** Same as the 5-min version's beat 2.
4. **(5:00–7:00) Chat demo: a real answer.** Same as 5-min beat 3, but pause to open `yingqiangyuan_me/prompts/knowledge-base.md` and show the structured résumé.
5. **(7:00–9:00) The Mixin pattern.** Open `yingqiangyuan_me/one/`. Show `one_00_main.py`, `one_01_config.py`, `one_02_boto3.py`. **Say:** "Each ~20 lines. Cross-cutting concerns are separate files. `cached_property` means the boto3 client isn't built until first use — meaningful for cold-start latency on serverless."
6. **(9:00–11:00) The format adapter.** Open `yingqiangyuan_me/ai_sdk_adapter.py`. Show `ai_sdk_message_generator`. **Say:** "This is the contract between the Vercel AI SDK and Bedrock. The event names — `text-start`, `text-delta`, `text-end`, `finish`, `[DONE]` — are the v5 spec, exact match. If they drift, the chat silently breaks. This is the first place I'd write tests."
7. **(11:00–13:00) The honest part — what's missing.** **Say:** "Three things I'd ship next, in priority order. (1) The frontend sends a FingerprintJS header to enable rate limiting; the backend logs it but doesn't enforce yet — that's the first thing. (2) Observability: I'm `rich.print`-ing token usage to stderr today; that needs to be CloudWatch metrics. (3) The test directories are empty. I know — that's the first thing a senior reviewer notices."
8. **(13:00–15:00) Tools & DX.** Open `mise.toml`. **Say:** "One file pins Python, Node, uv, pnpm versions and defines every task. `mise run dev` kills stale processes, then starts both servers under `concurrently`. I wrote `kill-dev-servers.py` because port-stuck dev servers were burning hours. The `dev` task `depends = ['kill']` so it's invisible most of the time." Close with the live URL again.

## Recovery moves — if something goes wrong live

| Failure mode | Recovery |
| :----------- | :------- |
| The chat returns an error or hangs | "Looks like the backend is cold-starting / Bedrock is throttling — let me show the architecture instead." Pivot to opening `01-knowhow-inventory.md`. **Don't** spend more than 30s waiting; the tension kills the demo. |
| The browser refuses to load the live URL | Have `mise run dev` already running locally as a fallback. Switch to `localhost:3000`. |
| Audience asks a question you can't answer | "Honest answer — I don't know. Let me check." Don't fabricate. Then either (a) read the code live to find out, or (b) note it and move on. |
| You accidentally open a `do-NOT-show` file | Close it without commentary. **Don't** narrate it ("oh that's nothing"). Just close and continue. |
| The live URL works but the chat is slow | "First request is cold — second one will be fast." Type a fast follow-up to demonstrate the cache hit. |
| The audience challenges the topic-lock by asking something cleverly off-topic | If the model refuses correctly: ✅ great signal. If it slips: "That's a known edge — system prompts aren't bulletproof; my next iteration adds an input-classifier as a backstop." Honesty wins. |

## Rehearsal questions — what a viewer might ask

Use these to drill the user before going live.

1. **"How much does it cost to run a single chat conversation?"** — Strong answer: "About `$X` for a 5-turn chat — Nova Micro is roughly `$0.000035` per 1k input tokens, knowledge base is cached after turn 1, so most of the cost is output. I'd put a hard cost cap at `$Y/mo` once observability is in." Weak answer: "I haven't measured."
2. **"What stops someone from spamming the chat to drain your AWS bill?"** — Strong answer: "Today, nothing — the FingerprintJS header is plumbed but not enforced. That's the first thing I'd ship: server-side token-bucket rate limiting." Weak answer: "FingerprintJS handles it." (No — that's the *plumbing*, not the enforcement.)
3. **"How do you handle the case where Bedrock returns a refusal of its own?"** — Strong answer: "The system prompt template handles model-generated refusals already. I distinguish them from API errors by checking `response.stopReason`. For API errors I'd surface a 502 / 503 to the frontend; for content refusals the message just streams as-is."
4. **"Why didn't you use OpenAI?"** — See `05-interview-playbook.md#Q3.1`.
5. **"What's the test coverage?"** — Strong answer: "Zero — and that's my top priority. The two highest-leverage targets are `ai_sdk_adapter.py` (the front-back contract) and `multi_round_bedrock_runtime_chat_manager.py` (mutable history). Snapshot tests for the first, mocked Bedrock for the second." Weak answer: "I haven't gotten to it yet." (Both are accurate, but the first signals you've thought about *what* to test, not just that you should.)

## Last beat — the close

Always close on a forward-looking note, not a retrospective one. Two options that both work:

- **Roadmap close.** "What I'd ship next, in priority order: server-side rate limiting on the FingerprintJS header, CloudWatch-based observability for token usage, and a real test suite. The architecture is ready for all three — the work is mostly *enforcement* of design decisions already made."
- **Pivot close.** "If you'd like to see how I'd extend this — say, add a tool-using agent that can search GitHub for my code samples — happy to whiteboard that next."

Never close with "and that's it" or "any questions?". Always end with a hook.
