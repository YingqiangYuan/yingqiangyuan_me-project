# 03 — Elevation Roadmap

> **Authoritative cross-references**: see 01-knowhow-inventory.md for current-state file paths.

This is the "if you had 3–6 more months on this project" doc. Every upgrade below is honest — none of them say "the project is bad". They say "the next maturity level looks like this". Where evidence in the code suggests a feature was *considered* and *deferred*, this doc names that explicitly.

## Executive view — top 5 highest-leverage upgrades

1. **Server-side rate limiting on `X-Client-Fingerprint`.** The header is sent by the frontend (`components/chat/chat.tsx`) and logged by the backend, but there is **no enforcement**. A single curl loop can drain Bedrock spend overnight. This is the highest-ROI hardening on the project.
2. **Observability and cost tracking for Bedrock.** Today, token usage is `rich.print`-ed to stderr by `ChatSession.debug_response()`. Without structured logs, request IDs, or a metrics sink, you can't audit cost, detect abuse, or even know when a Bedrock outage hit. Vercel + CloudWatch is the cheap path; Sentry-free-tier covers errors.
3. **A real test suite.** `tests_python/` and `tests_node/` are referenced in `mise.toml` but **don't exist**. The two highest-risk modules to cover are `ai_sdk_adapter.py` (it's the contract between the frontend SDK and Bedrock) and `multi_round_bedrock_runtime_chat_manager.py` (mutable history, easy to corrupt).
4. **Re-enable message validation.** `api/index.py` lines ~78–91 are commented out. Re-enabling them (and adding sane bounds) closes the "send a 100KB prompt" abuse path that shows up paired with #1.
5. **Decide on persistence: build it or remove it.** PynamoDB is declared in `pyproject.toml` but unused. Either commit (per-fingerprint chat history with a TTL) or delete (drop both packages, simplify `Boto3Mixin`). The half-state is the worst state — it suggests work that doesn't exist and confuses every reader.

## Upgrade areas

Each area below has the same structure: current state → target state → 2 alternatives → knowledge prerequisites → suggested learning path.

### Observability and ops readiness

**Current state.** Debug output is `print(..., file=sys.stderr)` via `yingqiangyuan_me/utils.py:debug` and `rich.print` from `ChatSession.debug_response()`. No structured JSON logging. No request correlation IDs. No metrics sink for token counts. The only health check is `GET /api/hello`. FingerprintJS headers are logged by `ai_sdk_adapter.debug_ai_sdk_request()` but never correlated with anything.

**Senior-engineer target state.**

- A request correlation ID (UUID) generated in `api/index.py` and propagated into every log line in the request scope (FastAPI middleware).
- Structured JSON logging via stdlib `logging` + `python-json-logger` (or rich's `JSON`-mode handler) → stdout → CloudWatch ingestion (Vercel does this natively).
- Token usage from each `client.converse(...)` response published to CloudWatch Metrics under a custom namespace (e.g. `yingqiangyuan/bedrock/{input_tokens, output_tokens, cache_read_tokens, cache_write_tokens}`).
- A monthly Bedrock spend alarm in CloudWatch tied to the chosen budget (e.g. $25/mo).
- Sentry (free tier is enough for a portfolio site) for unhandled-error capture.

**Alternatives.**

| Option | Pros | Cons |
| :----- | :--- | :--- |
| Native: CloudWatch Logs/Metrics + Sentry free | Zero new vendors; native Bedrock integration; minimal cost | CloudWatch query language is painful; no built-in latency dashboards |
| Third-party: Datadog or New Relic | Better UX, dashboards out of the box, distributed tracing | $50–200/mo; vendor lock-in; overkill for a portfolio |

**Recommendation.** Native option. The traffic profile of a personal portfolio doesn't justify a paid APM.

**Knowledge prerequisites.**

- AWS CloudWatch Logs ingestion + retention model.
- Structured logging best practices (12-factor logs, correlation IDs).
- Bedrock Converse API response shape (`ResponseMetadata.RequestId`, `usage.inputTokens`, `usage.cacheReadInputTokens`, `usage.cacheWriteInputTokens`).
- Vercel function log ingestion (it pipes stdout/stderr to its own log store; CloudWatch is opt-in via Lambda destinations).

**Learning path.**

1. AWS CloudWatch Logs basics — official AWS docs (10 min).
2. Python `logging` stdlib + `python-json-logger` quickstart (30 min hands-on).
3. Bedrock Converse `ResponseMetadata` and `usage` field reference (15 min).
4. Sentry FastAPI integration guide (15 min).

### Test depth and coverage

**Current state.** Zero test files. `tests_python/` and `tests_node/` directories don't exist on disk. `mise run test-python` / `test-node` and `pytest` (in `[project.optional-dependencies].test`) are wired and waiting.

**Senior-engineer target state.**

- ≥80% line coverage on `yingqiangyuan_me/` (skip thin re-export shims like `api.py`).
- The two contract-critical modules covered first:
  - `ai_sdk_adapter.py` — snapshot tests covering each AI SDK v5 → Bedrock conversion case (text-only message, multi-part message, system-message-skipping). Mock with `moto` if AWS isn't needed.

- One frontend test for the fingerprint-injection fetch wrapper in `components/chat/chat.tsx`.

**Alternatives.**

| Option | Pros | Cons |

| pytest-cov + Vitest for the JS side | Better reports; snapshot testing; widely used | Adds deps; another config file; slight migration cost |

**Recommendation.** Stay with the wired tooling. Add `pytest-cov` for HTML reports if/when coverage becomes a number you publicly track.

**Knowledge prerequisites.**

- pytest fixtures, `parametrize`, `monkeypatch`.
- `moto` library for mocking AWS service calls (especially Bedrock Runtime).
- The AI SDK v5 Data Stream Protocol event spec (so snapshot tests don't drift).
- Node's built-in test runner + `node:assert` (no Jest).

**Learning path.**

1. pytest fixtures + parametrize — Real Python tutorial (20 min).
2. `moto` Bedrock Runtime mocking — moto docs + 1 example (15 min).
3. Vercel AI SDK v5 protocol spec — official docs (15 min).
4. Node `--test` runner basics — Node official docs (10 min).

### Performance and scalability

**Current state.** Bedrock is called synchronously inside the FastAPI handler. Prompt caching via `cachePoint` is enabled (good), but the cache hit rate isn't tracked. `paths.py` caches `instruction.md` and `knowledge-base.md` content via `@cached_property` so the prompt files only hit disk once per cold start (good). No request timeout on the Bedrock client. No retry/backoff on the frontend. Frontend `useChat` is the AI SDK default — single-shot, no resilience layer.

**Senior-engineer target state.**

- p99 chat latency SLO < 8s (Vercel hard timeout is 10s on hobby, 60s on Pro — leaves headroom).
- Cache hit rate ≥ 70% in production (validate via `cacheReadInputTokens` / `inputTokens` ratio in CloudWatch).
- Cold-start budget < 2s (measure via Vercel duration metrics; lazy-import slow modules if needed).
- Frontend retry: 3 attempts with exponential backoff (250ms, 500ms, 1s), only on 5xx/network errors, never on 4xx.
- Bedrock client timeout set explicitly (e.g. 25s read, 5s connect) so a hanging call doesn't burn the whole Vercel quota.

**Alternatives.**

| Option | Pros | Cons |
| :----- | :--- | :--- |
| Stay synchronous, optimise cold start (lazy imports, smaller deps) | Simple; the chat UX feels live | Doesn't help if Bedrock itself is slow; still capped by Vercel timeout |
| Move to a queue (SQS + worker) for long generations | Decouples latency; survives timeouts | Breaks SSE streaming UX; massive architectural change for marginal gain at this traffic |

**Recommendation.** Stay synchronous. Add measurement first; only optimise once you have data.

**Knowledge prerequisites.**

- Vercel Serverless Function execution model — cold start anatomy (function package size, init time, runtime version).
- Bedrock Converse `cachePoint` mechanics and how prompt caching is billed.
- SSE streaming + browser timeout behavior (browsers and Vercel both have idle timeouts).
- Load testing with `k6` or `Artillery` against streaming endpoints.

**Learning path.**

1. Vercel cold start guide (10 min).
2. Bedrock prompt caching cost docs (15 min).
3. `k6` HTTP streaming example (20 min hands-on).
4. CloudWatch custom metric publication via boto3 (10 min).

### Security and threat model

**Current state.** No auth (intentional — public chatbot). No CORS restrictions explicitly set. No rate limit. Message length validation in `api/index.py` is commented out. FingerprintJS produces a `X-Client-Fingerprint` header that is **logged but not enforced**. System prompt has good jailbreak guardrails (`prompts/instruction.md` lines 75–82) but those are model-level, not runtime.

**Threats worth modeling.**

| Threat | Likelihood (today) | Impact | Notes |
| :----- | :----------------- | :----- | :--- |
| Unmetered API abuse (curl loop) | High | High ($-cost) | No rate limit. Most urgent. |
| Prompt injection (extracting hidden context) | Medium | Low–Medium | System prompt is robust but not bulletproof. |
| 1MB-message DoS | Medium | High (cost + timeout) | Validation commented out. |
| Fingerprint spoofing | High once #1 fixed | Mitigates rate limit | Plan for it; consider IP+fingerprint composite. |
| Cross-session cache leakage in Bedrock | Very low | Theoretical | Bedrock cache is account-scoped; not a realistic risk. |

**Senior-engineer target state.**

- Token-bucket rate limit keyed on `X-Client-Fingerprint` (e.g. 10 req/min, 100 req/day per fingerprint). Bucket state in DynamoDB or, for a single-region single-replica deploy, an in-process LRU.
- Re-enabled message length validation: max 5,000 chars. Reject anything longer with `HTTP 413`.
- Optional: simple prompt-injection regex rejection (e.g. block `ignore (previous|all) instructions`). Cheap, catches script kiddies, doesn't catch sophisticated attackers.
- Bedrock throttling (`ThrottlingException`) handled gracefully → return `HTTP 429` to client, frontend already has toast logic ready.
- A monthly Bedrock spend hard cap via CloudWatch alarm + Lambda that disables the IAM principal at $X/mo.

**Alternatives for rate limiting.**

| Option | Pros | Cons |
| :----- | :--- | :--- |
| In-memory token bucket | Zero latency, zero deps | Lost on Vercel cold start; broken across multiple replicas |
| DynamoDB-backed (e.g. with PynamoDB which is *already in deps*) | Survives cold starts; per-fingerprint audit trail; ties into the "remove or commit" decision in #5 | Extra latency per request; DynamoDB minimum cost (~$1/mo at this volume) |

**Recommendation.** DynamoDB-backed — it's the natural place to actually use the dependency that's already declared. This converts the "PynamoDB unused" problem into a "PynamoDB used for the first thing it should have been".

**Knowledge prerequisites.**

- Token bucket algorithm and sliding-window variants.
- OWASP LLM Top 10, especially LLM01 (prompt injection) and LLM04 (unbounded consumption).
- Bedrock throttling and `RetryConfig` in boto3.
- DynamoDB conditional writes (the safe atomic increment for token-bucket state).

**Learning path.**

1. Token bucket algorithm explanation (10 min, any blog post).
2. OWASP LLM01 + LLM04 (15 min).
3. PynamoDB quickstart with conditional writes (30 min).
4. Bedrock throttling and `RetryConfig` docs (15 min).

### API / contract design

**Current state.** Frontend uses `@ai-sdk/react`'s `useChat`. Backend parses with `vercel-ai-sdk-mate.RequestBody(**request_body_data)`. Response is hand-emitted SSE matching AI SDK v5. No OpenAPI spec is published. No version negotiation header — if the frontend `ai` package ever upgrades to v6 with a different protocol, the backend will break silently.

**Senior-engineer target state.**

- Pydantic v2 `BaseModel` for the request body (or trust `vercel-ai-sdk-mate`'s typed dataclasses, but assert the contract).
- A `Content-Type: application/vnd.yqy.v1+json` (or simpler: a `?protocol=v1` query parameter) version-tag, with the backend rejecting unknown versions.
- FastAPI `responses={400, 413, 429, 502, 504}` declarations on `/api/chat` so the OpenAPI it auto-generates is accurate.
- A single contract test that asserts the SSE event sequence emitted by `ai_sdk_message_generator()` matches the AI SDK v5 spec — pin one of these as the canary.

**Alternatives.**

| Option | Pros | Cons |
| :----- | :--- | :--- |
| Pydantic + FastAPI's auto-OpenAPI | Free, no extra tools | Manual `responses` declarations |
| Full OpenAPI 3.1 spec source-of-truth + codegen | Single SoT; contract-driven | Heavy for a one-route public API |

**Recommendation.** Pydantic + FastAPI auto-OpenAPI. Don't over-engineer for one route.

**Knowledge prerequisites.**

- Vercel AI SDK v5 Data Stream Protocol spec.
- Pydantic v2 strict mode.
- FastAPI `responses=` and OpenAPI tags.
- SSE semantics: keep-alive, error mid-stream, `[DONE]` sentinel.

**Learning path.**

1. AI SDK v5 protocol spec (15 min).
2. Pydantic v2 strict mode + JSON schema gen (20 min).
3. FastAPI custom `responses=` examples (10 min).
4. Snapshot testing for streaming protocols (15 min).

No CI gating before deploy. No rollback procedure documented (Vercel UI's "Promote a previous deployment" is the de facto rollback). `vercel.json` has `"crons": []` — empty placeholder.

### Data layer (PynamoDB unused — decide)

**Current state.** `pynamodb` (>=6.1) and `pynamodb-session-manager` (>=0.1.2) are declared in `pyproject.toml`. `Boto3Mixin.bsm` (`yingqiangyuan_me/one/one_02_boto3.py`) is wired specifically for PynamoDB. No PynamoDB models are defined; nothing imports it. There is no chat history persistence. Frontend `chatId` is hardcoded `"001"`.

**The fork in the road.**

- **Path A: build it.** Add a single PynamoDB model `Session(pk=fingerprint, sk=created_at_isoformat)` storing the message list as JSON, with DynamoDB TTL of 7 days. Frontend persists `chatId` to `localStorage` and offers a "Continue last conversation" button. This kills two birds: it gives chat history (a real product feature) and provides the storage backend for the rate-limit token bucket.
- **Path B: delete it.** Remove `pynamodb` and `pynamodb-session-manager` from `pyproject.toml`, simplify `Boto3Mixin` to drop `bsm`, accept that the chat is stateless. Cleaner, smaller, less to explain.

**Recommendation.** Path A *if* you also do server-side rate limiting (#1 above) — the same DynamoDB infrastructure serves both. Path B *if* you're not going to do rate limiting in the next iteration — the unused dep is just noise.

**Knowledge prerequisites.**

- PynamoDB model definition, query patterns.
- DynamoDB TTL.
- Privacy considerations for storing chat history (even pseudonymous-by-fingerprint chats can be PII-adjacent if users paste real names/details).

**Learning path.**

1. PynamoDB quickstart (20 min).
2. DynamoDB TTL semantics (10 min).
3. Privacy/PII basics for chat applications (15 min).

## Projects-not-included

These are features that the code shows evidence of being *considered and deferred*, not oversights. Each is stated as a hypothesis with the evidence; the inheritor should confirm with the project owner before acting.

1. **PynamoDB-backed session storage** — Evidence: `pynamodb` + `pynamodb-session-manager` declared in `pyproject.toml`; `Boto3Mixin.bsm` documented for PynamoDB use; zero models defined. **Hypothesis:** considered for chat history; deferred to ship the MVP.
2. **Message length / content validation** — Evidence: `api/index.py` lines ~78–91 are commented out; `get_last_user_message_text()` already exists in `ai_sdk_adapter.py` for this purpose. **Hypothesis:** prototyped, then disabled (perhaps because users were legitimately pasting long résumés during testing).
3. **Server-side rate limiting via FingerprintJS** — Evidence: `X-Client-Fingerprint` is sent and logged; frontend toast logic detects `429` / `"Too many requests"` / `"Rate limit"`; backend never enforces. **Hypothesis:** the entire client-side instrumentation was built first; server-side enforcement was deferred.
4. **Multi-session chat / conversation resume** — Evidence: `chatId = "001"` is hardcoded in `components/chat/chat.tsx`; no `localStorage` for it; PynamoDB unused. **Hypothesis:** initial design considered "continue your previous chat" UX; deferred along with PynamoDB.
5. **Tool use / Bedrock agents** — Evidence: `client.converse(...)` is called with no `toolConfig`; the backend walkthrough explicitly states no tools are used. **Hypothesis:** intentionally out of scope — tools would balloon scope (auth, error handling, per-tool limits) and the use case ("introduce YingQiang") doesn't need them.
6. **Light-mode theme toggle** — Evidence: `next-themes` is in `package.json`; `<html>` is hardcoded `dark` in `app/layout.tsx`; no `<ThemeProvider>` is rendered. **Hypothesis:** wired so a future light-mode shipper has the dependency present, but the toggle is intentionally not in v1.
7. **Multi-modal input** — Evidence: the input component is named `multimodal-input.tsx`; only text parts are accepted today (`ai_sdk_adapter.part_to_bedrock_content` only handles `type == "text"`). **Hypothesis:** named for future expansion (image upload, file).
8. **Scheduled tasks** — Evidence: `vercel.json` has `"crons": []` (empty placeholder). **Hypothesis:** placeholder for a future daily-cost-summary cron.
