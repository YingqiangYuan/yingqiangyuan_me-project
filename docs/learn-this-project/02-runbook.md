## Prerequisites

Versions are pinned in `mise.toml`. If you have `mise` installed, `mise install` will fetch them all automatically.

| Tool | Required version | How to get it |
| :--- | :--------------- | :------------ |
| **mise** | latest | macOS: `brew install mise`. Other: see https://mise.jdx.dev/installing-mise.html |
| **Python** | 3.12 | `mise use python@3.12` (mise will install) |
| **Node.js** | 24 | `mise use node@24` |
| **uv** | latest | `mise use uv@latest` (Python package manager ‚Äî pip replacement) |
| **pnpm** | 10.33.3 | `mise use npm:pnpm@10.33.3` (faster npm) |
| **Claude Code** | latest | optional but pinned in `mise.toml` (`claude = "latest"`) |

After installing mise, run `mise install` once at the project root ‚Äî it reads `mise.toml` and installs everything else.

You also need:

- **AWS credentials** with `bedrock:InvokeModel` permission for `us.amazon.nova-micro-v1:0`. Either set them in `~/.aws/credentials` (a profile picked up by `boto3`'s default credential chain) or put `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` in `.env` at the project root.
- **A `.env` file**. Copy `.env.example` and fill it in:

  ```bash
  cp .env.example .env
  # edit .env and add your AWS_ACCESS_KEY_ID / AWS_SECRET_ACCESS_KEY
  ```

## First-time setup

Run from the project root, in this order:

```bash
# 1. Pin tool versions and install (one-time, idempotent)
mise install

# 2. Create the Python virtualenv at .venv/
mise run venv-create

# 3. Install all dependencies (Python + Node)
mise run inst
```

Verify the install:

```bash
.venv/bin/python -c "from yingqiangyuan_me import one; print(one.config.aws_region)"
# expect: a region string (e.g. "us-east-1") ‚Äî proves AWS creds load and the singleton wires up

pnpm exec next --version
# expect: 16.x ‚Äî proves Next is installed
```

## Day-to-day commands

All routine work goes through `mise run <task>`. Tasks live in `mise.toml`.

| Task | Command (verbatim) | What it does |
| :--- | :----------------- | :----------- |
| Start dev | `mise run dev` | Kills stale servers, then starts Next.js (3000) + FastAPI (8000) under `concurrently`. |
| Kill stale servers | `mise run kill` | Calls `kill-dev-servers.py`. Use when ports are stuck. |
| Run all tests | `mise run test` | `test-python` + `test-node`. |
| Python tests only | `mise run test-python` | `pytest tests_python` |
| Node tests only | `mise run test-node` | `node --test tests_node/*.test.js` |
| Install deps | `mise run inst` | Python + Node, parallel. |
| Install Python only | `mise run inst-python-deps` | `uv sync --all-extras` |
| Install Node only | `mise run inst-node-deps` | `pnpm install` |
| Recreate venv | `mise run venv-remove && mise run venv-create && mise run inst-python-deps` | When `.venv` gets wedged. |
| Export `requirements.txt` | `mise run export` | For Vercel build. Generates `requirements.txt` from `pyproject.toml`. |
| Next dev only | `mise run next-dev` | Just Next.js on `:3000`. |
| FastAPI dev only | `mise run fastapi-dev` | Just FastAPI on `:8000` (uvicorn with `--reload`). |

> ‚öÝÔ∏è `mise run dev` and `npm run dev` are **not equivalent**. The `npm` version uses `pip3 install -r requirements.txt && python3 ...`; the `mise` version uses `uv` and the `.venv`. **Use `mise run dev` locally.**

## Running locally

The dev experience is one command:

```bash
mise run dev
```

This will:

1. Run `mise run kill` first (the `dev` task `depends = ["kill"]` in `mise.toml`).
2. Start Next.js on **http://localhost:3000** (with hot reload).
3. Start FastAPI on **http://127.0.0.1:8000** (uvicorn with `--reload`).
4. Run them under `concurrently` so logs are interleaved with prefix labels.

What to open in the browser:

- **Landing page** ‚Äî http://localhost:3000/
- **Chat** ‚Äî http://localhost:3000/chat
- **API health** ‚Äî http://localhost:3000/api/hello (proxied through Next; or http://127.0.0.1:8000/api/hello direct)
- **API chat** ‚Äî `POST http://localhost:3000/api/chat` (used by the frontend; rarely hit directly)

How the proxy works: `next.config.js` rewrites `/api/*` from `:3000` to `http://127.0.0.1:8000/api/*` in dev. In prod, Vercel serves `/api/*` directly from the Serverless Functions in `api/` ‚Äî no rewrite needed.

To stop: `Ctrl-C` in the terminal. If the processes hang or orphan (you'll know ‚Äî port 3000 stays bound), run `mise run kill`.

## Running each side independently

Sometimes you want to debug just one side. Use two terminals:

```bash
# Terminal 1 ‚Äî Next.js
mise run next-dev

# Terminal 2 ‚Äî FastAPI
mise run fastapi-dev
```

The full FastAPI command (in case you need to tweak flags) is:

```bash
uv run -- python -m uvicorn api.index:app --reload --host 127.0.0.1 --port 8000
```

## Tests

Test infrastructure is wired but **the test directories are empty**. There are no test files yet. The mise tasks expect `tests_python/` and `tests_node/` directories to exist:

- `mise run test-python` runs `.venv/bin/pytest tests_python` ‚Äî uses the `pytest` extra from `pyproject.toml`'s `[project.optional-dependencies]`.
- `mise run test-node` runs `node --test tests_node/*.test.js` ‚Äî uses Node's built-in test runner (no Jest, no Vitest).

When you write your first tests:

```bash

```

Suggested coverage priority (from 03-elevation-roadmap.md):

## Common failures and fixes

| Symptom | Likely cause | Fix |
| :------ | :----------- | :-- |
| Port 3000 / 8000 already in use | Previous `mise run dev` orphaned its children | `mise run kill` (or just rerun `mise run dev` ‚Äî it kills first) |
| `ImportError: No module named yingqiangyuan_me` | Dev server invoked outside the `.venv` | Use `mise run fastapi-dev`, never plain `uvicorn ...` |
| `botocore.exceptions.NoCredentialsError` | `.env` missing or `~/.aws/credentials` not configured | Copy `.env.example` to `.env` and fill in keys, or run `aws configure` |
| `AccessDeniedException: ... bedrock:InvokeModel ...` | IAM principal doesn't have Bedrock access | Add `bedrock:InvokeModel` policy for the inference profile region. Note: `us.amazon.nova-micro-v1:0` is a **cross-region inference profile**; permission must allow the full set of regions in the profile. |
| Chat shows "thinking..." forever, no response | FastAPI not running, or `next.config.js` rewrite broken | Check `mise run dev` output for FastAPI errors; verify http://127.0.0.1:8000/api/hello returns OK |
| `pnpm install` errors about workspaces / corepack | Corepack disabled | `corepack enable` once globally; `vercel.json` already enables it for Vercel builds |
| Next.js build error about font fetching | No internet at build time | Google fonts are fetched at build; if you build offline use `next/font/local` instead. |
| `next dev` hot reload doesn't refresh `prompts/*.md` changes | `paths.py` caches the prompt content via `cached_property` | Restart `mise run fastapi-dev`. (This is by design ‚Äî see 01 inventory.) |
| Vercel deploy fails on Python | `requirements.txt` stale | Run `mise run export` and commit the updated file. Vercel uses `requirements.txt`, not `pyproject.toml`. |
| `mise run kill` says "no processes found" but ports still bound | Edge case ‚Äî `kill-dev-servers.py` PID detection missed | The script also tries `lsof -ti :3000` etc. as a fallback. If still stuck: `lsof -ti :3000 \| xargs kill -9` |

If you hit something not on this list, add it here when you fix it ‚Äî that's how this section grows.

## Where to look when something breaks

| Concern | Where to look |
| :------ | :------------ |
| Backend errors / Bedrock errors | FastAPI's stderr (your terminal in dev; Vercel function logs in prod). The `debug()` helper in `yingqiangyuan_me/utils.py` writes to stderr, so debug prints land here. |
| Frontend errors | Browser console + Network tab. SSE responses appear as a single ongoing request ‚Äî open it to see the event stream. |
| Bedrock token usage | Currently logged via `rich.print` from `ChatSession.debug_response()` ‚Äî appears in FastAPI stderr. Not yet exported to a metrics store (see `03-elevation-roadmap.md#observability`). |
| Vercel deploy logs | Vercel dashboard ‚Üí project ‚Üí Deployments ‚Üí latest ‚Üí "Build Logs" + "Function Logs". |
| AWS Bedrock service status | https://health.aws.amazon.com/health/status (filter to your region) |
| Stuck dev processes | `ps aux \| grep -E "next dev\|uvicorn\|concurrently"` then `kill -9 <pid>`, or `mise run kill` |
| `kill-dev-servers.py` what-it-killed log | The script prints what it killed to stdout ‚Äî scroll up in your terminal |
| Cold-start latency | Vercel function logs show duration. Cold starts are typically 1.5‚Äì3s; warm calls add Bedrock latency only (~500ms‚Äì2s). |

## Deploy

Production deploys to Vercel automatically on push to the `main` branch (assuming the repo is connected to a Vercel project). The build pipeline:

1. Vercel reads `vercel.json` (Python 3.12, corepack enabled).
2. Runs the Next.js build (`pnpm build` implicitly).
3. Detects `api/index.py`, packages Python deps from `requirements.txt`, deploys as a Serverless Function.
4. Routes `/api/*` to the function; everything else to the static/SSR Next.js output.

Required Vercel project env vars (set in the Vercel dashboard, not in `.env`):

- `AWS_ACCESS_KEY_ID`
- `AWS_SECRET_ACCESS_KEY`
- (optional) `AWS_REGION` if not the default

If you change Python deps in `pyproject.toml`, run `mise run export` and commit the updated `requirements.txt` ‚Äî Vercel does **not** read `pyproject.toml` directly.

There is no staging environment today. Vercel preview URLs (auto-generated for non-`main` branches) act as ad-hoc staging ‚Äî they get the same Vercel env vars by default.
