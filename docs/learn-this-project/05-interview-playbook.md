## Opening — the 90-second pitch (model script)

**Model pitch (~150 words):**

> "yingqiangyuan_me is my personal portfolio site at yingqiangyuan.me — a Next.js marketing page paired with an AI chatbot that answers career questions about me, and only me. ## Round 1 — what you have (mechanism, data flow, architecture)

### Q1.1 — "Walk me through what happens when a user types a message and presses Enter."

**Model answer.** The frontend's `useChat` hook (from `@ai-sdk/react`) POSTs to `/api/chat` with the AI SDK v5 message format and a `X-Client-Fingerprint` header injected by a wrapped `window.fetch`. Next.js rewrites that path in dev to `127.0.0.1:8000`; in prod, Vercel routes it directly to the Serverless Function. FastAPI's handler in `api/index.py` parses the body via `vercel-ai-sdk-mate`, converts SDK messages to Bedrock format with `ai_sdk_adapter`, constructs a `ChatSession`, and calls `bedrock_runtime_client.converse()`. The response is streamed back as SSE events (`text-start` → `text-delta` × N → `text-end` → `finish` → `[DONE]`) which `useChat` re-assembles into rendered text.

**Follow-ups.** "Why streaming SSE not WebSockets?" "What if the user closes the tab mid-stream — what happens server-side?"

**Weak-answer signature.** Skips the SDK→Bedrock format conversion (sign that the candidate didn't actually understand `ai_sdk_adapter.py`) or doesn't mention the fingerprint header at all.

### Q1.2 — "Where does the AI's knowledge of you come from? Walk me through how the chatbot stays factual."

**Model answer.** Two files: `prompts/instruction.md` (system prompt with topic constraints + jailbreak protection) and `prompts/knowledge-base.md` (structured résumé). The instruction is set as Bedrock's `system` parameter. The knowledge base is injected as the **first user turn** of the conversation, paired with a `cachePoint` — Bedrock then caches that prefix across subsequent turns, which both grounds the model and minimises cost. Both files are read once via `paths.py:cached_property` so they don't hit disk on every request.

**Follow-ups.** "What's the cost difference between system-prompt-injection and first-turn-injection?" "How would you keep this in sync with my real résumé?"

**Weak-answer signature.** Says "the system prompt has all the info" — that's wrong; the system prompt is the *rules*, the knowledge base is the *facts*.

### Q1.3 — "Why is the backend Python instead of just using Next.js Route Handlers?"

**Model answer.** The AWS/Bedrock SDK ecosystem is more mature in Python — `boto3-dataclass[bedrock-runtime]` gives you typed responses, `boto-session-manager` handles credential paths cleanly, and `vercel-ai-sdk-mate` provides typed SDK protocol parsing. Doing this in TS would either mean re-implementing those, or living with raw `aws-sdk` calls and untyped streaming protocol parsing. The boundary is also clean: Next does what Next is good at (UI, SSR, image optimisation); Python owns AI plumbing.

**Follow-ups.** "What's the cost of having two runtimes — bundle size, deployment complexity?" "Would you do it the same way today?"

**Weak-answer signature.** "Because Python is better for AI." That's not a tradeoff, it's a vibe.

### Q1.4 — "What is the `One` class and why does it exist?"

**Model answer.** It's a singleton (`one`, exposed at three import paths for stability) composed of mixins — `ConfigMixin` provides `self.config`, `Boto3Mixin` provides AWS clients. Each mixin is ~20 lines; each provides its responsibilities as `@cached_property` so AWS clients aren't built until first use (which keeps cold-start cheap). Adding a new cross-cutting concern — DynamoDB, Bedrock Agents, secrets manager — is a one-file change: add `one_03_db.py`, add the mixin to `One`'s base classes, done.

**Follow-ups.** "Why mixins instead of dependency injection?" "What's the failure mode if two mixins want a property with the same name?"

**Weak-answer signature.** "It's just a singleton." Misses the composition story.

### Q1.5 — "Tell me about the SSE response generator."

**Model answer.** `ai_sdk_message_generator` in `ai_sdk_adapter.py` yields a fixed sequence of SSE event strings that match the AI SDK v5 Data Stream Protocol exactly: `text-start` (announces the assistant turn), repeated `text-delta` (the actual streamed tokens), `text-end`, `finish` (carries finish reason + usage), and `[DONE]` (sentinel). FastAPI wraps this in a `StreamingResponse` with the `x-vercel-ai-ui-message-stream` header (the v5 spelling — using the v4 `x-vercel-ai-data-stream` would silently break the frontend parser).

**Follow-ups.** "What if the network drops mid-stream — does the frontend recover?" "How do you test this?"

**Weak-answer signature.** Doesn't mention the v5 header name at all (spec adherence is the point of the question).

### Q1.6 — "How does the project know whether it's running locally vs on Vercel?"

**Model answer.** `runtime.py` checks `os.environ.get("VERCEL", "NOTHING") == "1"` — Vercel sets `VERCEL=1`, anything else is local. `Config.new()` then dispatches: local calls `load_dotenv()` first (so `.env` is read), Vercel reads env vars directly (Vercel injects them natively). It's centralised — only `Config` ever branches on runtime; everything else is environment-agnostic.

**Follow-ups.** "What if you wanted to add a 'staging' environment — how would you change this?" "Could the same code run on AWS Lambda directly?"

**Weak-answer signature.** Lists every env-aware piece in the code — they're not, that's the point.

## Round 2 — what you'd elevate (3–6 more months)

### Q2.1 — "If you had three more months, what's the first thing you'd ship?"

**Model answer.** Server-side rate limiting on `X-Client-Fingerprint`. The frontend already sends it; the chat UI's toast already detects 429 errors; the backend just *doesn't enforce*. With no rate limit, a single curl loop can drain Bedrock spend in hours. I'd implement a token-bucket using PynamoDB (which is already in `pyproject.toml`, unused) — that converts an unused dependency into the storage backend for what's by far the highest-leverage hardening.

**Follow-ups.** "What rate would you set?" "What about IP-based rate limiting as a fallback?"

**Weak-answer signature.** "More features." A senior answer is operational, not feature-driven.

### Q2.2 — "How would you add observability without paying for a third-party APM?"

**Model answer.** Three pieces: structured JSON logging from `python-json-logger` to stdout (Vercel ingests it natively); a request correlation UUID propagated through every log line in the request scope; CloudWatch custom metrics for `inputTokens`, `outputTokens`, `cacheReadInputTokens` published per `client.converse(...)` response. Then a CloudWatch alarm at `$X/mo` Bedrock spend that fires at the first sign of abuse. Sentry's free tier handles unhandled exceptions. Total cost: a few dollars a month at this volume.

**Follow-ups.** "How do you handle correlation across the streaming response?" "What's the right log retention?"

**Weak-answer signature.** "Add Datadog." That's expensive and overkill for a portfolio.

### Q2.3 — "Test strategy: where do you start, and why?"

**Model answer.** Two highest-leverage targets first. (1) `ai_sdk_adapter.py` — it's the contract between the frontend SDK and Bedrock; if it silently breaks on a SDK version bump, the entire chat fails with no obvious symptom. Snapshot tests covering text-only, multi-part, system-message-skipping cases. (2) `multi_round_bedrock_runtime_chat_manager.py` — `_messages` is mutated in place; a regression that shares state across requests would be invisible until under load. After those, an integration test of `/api/chat` with a mocked Bedrock client (via `moto`) asserting the SSE event sequence.

**Follow-ups.** "Coverage % isn't a quality metric — what would you measure instead?" "How would you test the streaming behaviour end-to-end?"

**Weak-answer signature.** "Get to 80% coverage." Coverage is a floor, not a strategy.

### Q2.4 — "What's the right thing to do about PynamoDB?"

**Model answer.** Decide and act. Today it's in `pyproject.toml`, it's wired through `Boto3Mixin.bsm`, but no model is defined and nothing imports it — the worst possible state, because every reader has to ask "is this used?". Option A: build a `Session` model with TTL=7 days, persist chat history, and reuse the same DynamoDB table for the rate-limit token bucket — that justifies the dependency twice over. Option B: rip it out, simplify the mixin, accept that the chat is stateless. I'd pick A *if* I'm doing rate limiting in the same iteration, because the infrastructure pays for itself. B otherwise.

**Follow-ups.** "How would you handle privacy of stored chat history?" "What's the failure mode if DynamoDB is slow?"

**Weak-answer signature.** "Leave it for now." That's how repos rot.

## Round 3 — alternatives considered (the "why not X?" round)

### Q3.1 — "Why not use OpenAI or Claude API directly instead of Bedrock?"

**Model answer.** Bedrock gives me one-vendor billing (already inside AWS), prompt caching that's first-class, and access to the Nova model family which is significantly cheaper than GPT-4-class for short Q&A. The tradeoff is a slightly thinner SDK ecosystem and the cross-region inference profile complexity (`us.amazon.nova-micro-v1:0` is not a plain model ID). For a personal portfolio with cost sensitivity, that's the right choice. For a product where multi-vendor is a feature (failover, A/B), I'd pick differently.

**Follow-ups.** "Bedrock pricing changed — would you re-evaluate?" "What's the latency comparison?"

**Tradeoff matrix:** Bedrock vs OpenAI vs Anthropic-direct on (cost, latency, ecosystem maturity, vendor lock-in, billing complexity).

### Q3.2 — "Why not put the chat history in localStorage instead of planning a DB?"

**Model answer.** localStorage works for a single browser; it doesn't survive across devices, and it gives you no way to do per-user analytics or rate-limiting. For *this* project (anonymous public chatbot), localStorage might actually be the right answer — it's simpler, free, and the user owns their data. The reason I'd still consider DynamoDB is that the token-bucket for rate limiting needs a server-side store anyway, so adding session persistence on top is marginal cost.

**Follow-ups.** "What's the GDPR story for storing pseudonymous chats?"

### Q3.3 — "Why not enforce the rate limit in Vercel Edge Middleware instead of FastAPI?"

**Model answer.** Edge middleware runs before the function and wouldn't burn Function compute on a rejected request — strictly better. The reason I'd start in FastAPI is that the *fingerprint* logic has to be there anyway (the backend needs to log who hit the limit); Edge would still need to call FastAPI for that, or duplicate logic. The right architecture eventually is Edge for the cheap rejection, FastAPI for the expensive cases. But a senior should know "do the cheap version first; promote to Edge when traffic justifies it".

**Follow-ups.** "What if the fingerprint ever needs to be cryptographically validated?"

### Q3.4 — "Why not use `next-themes` to actually ship a light-mode toggle?"

**Model answer.** It's wired but not exposed because shipping it correctly means: theme provider, persisting user choice, server-rendering-safe, not flashing the wrong theme on first paint, and design work for a light palette. Each of those is small, but together they're a feature, not a toggle. Putting the dep in `package.json` now means a future shipper doesn't have to do a dependency PR. Not unfinished — *staged*.

**Follow-ups.** "How do you avoid the `next-themes` SSR-flash issue?"

### Q3.5 — "Why three levels of `one` re-export instead of one canonical import?"

**Model answer.** Public interface stability. `from yingqiangyuan_me import one` is what handlers should use; if I refactor internals tomorrow, that import doesn't change. `yingqiangyuan_me.api.one` is the named-public-API path. `yingqiangyuan_me.one.api.one` is the internal explicit path. Yes, three levels look redundant — and yes, you could collapse them today — but the cost of keeping them is zero, and the value if I ever rename `one/` is that no caller breaks.

**Follow-ups.** "When does adding indirection cost more than it saves?"

## Round 4 — problems hit (real bugs / debugging hypotheticals)

### Q4.1 — "Imagine the chat shows 'thinking…' forever and never replies. How do you debug?"

**Model answer.** Three checks in order. (1) Frontend network tab: is the SSE connection open? Is data flowing? If the request never returns, the backend is hanging. (2) FastAPI stderr: does the `debug_ai_sdk_request` log appear? If yes, the request reached the handler. Look for Bedrock errors. (3) Bedrock client behaviour: a hanging `client.converse()` with no read timeout could just sit forever. The fix is configuring a 25s read timeout on the boto3 client and surfacing `ThrottlingException` as a 429 to the frontend (which already has toast logic for it).

**Follow-ups.** "What if the request reaches the handler but no stderr log appears?" "How would observability have made this faster?"

### Q4.2 — "After editing `prompts/instruction.md`, the AI is still using the old prompt. Why?"

**Model answer.** `paths.py` caches the file content via `@cached_property`. The file is read once on first access and never re-read. In dev, restart `mise run fastapi-dev`. In prod, deploy. That's intentional: filesystem reads are cheap to amortise, but you don't want them on every request. The downside is exactly this confusion — worth a comment in the file or a "WHY restart" note in the runbook.

**Follow-ups.** "How would you make hot-reloading prompts work in dev only?"

### Q4.3 — "On Vercel, the first request after a quiet hour is slow but subsequent ones are fast. Why?"

**Model answer.** Cold starts. The Serverless Function container sleeps after idle; the next request pays for container init, Python interpreter start, package import, and `cached_property` initialisation of the Bedrock client. Subsequent requests reuse the warm container. Mitigations: keep the function bundle small (lazy-imports for heavy modules), pay for Vercel's "always warm" feature on paid plans, or accept the cost — for a portfolio site, a 2s first-request hit is acceptable.

**Follow-ups.** "How would you measure cold-start vs warm latency in prod?"

### Q4.4 — "Frontend toast says 'rate limited' but the backend logs no rejection. What's happening?"

**Model answer.** That's not a real bug today — it's a tell that the rate-limit code path isn't enforced. The frontend's toast detector matches any error containing `429`, `Too many requests`, or `Rate limit`. If a Bedrock `ThrottlingException` ever surfaces with one of those substrings, the toast fires *correctly* — but it's surfacing AWS-level throttling, not our limit. The fix when I implement rate limiting: distinguish "our 429" from "Bedrock's 429" with a `Retry-After` header and an explicit error code.

**Follow-ups.** "How would the user experience differ between those two cases?"

## Round 5 — production pushbacks (senior-level pressure)

### Q5.1 — "Traffic spikes 100x tomorrow. What breaks first?"

**Model answer.** Bedrock spend, immediately — there's no rate limit. Second: cold-start storms if Vercel scales out aggressively (each new container pays the init cost). Third: Bedrock rate limits at the AWS account level start throwing `ThrottlingException`, which the frontend sees as generic errors today (no per-rate-limit messaging). The mitigations in priority order are exactly the elevation roadmap's top items: rate limit, observability, throttling-aware retry. None of them require an architecture change.

**Follow-ups.** "Where would a queue help?" (Answer: it wouldn't, for SSE chat — it'd break the streaming UX.)

### Q5.2 — "A P0 fires now: chats are returning empty replies for everyone. You have 5 minutes. What do you check?"

**Model answer.** Vercel function logs first — is the handler running? If yes, look for Bedrock errors. If `ThrottlingException`: AWS quota issue, page AWS support, fall back to a "we're experiencing high demand" message client-side. If `AccessDeniedException`: credentials rotated and Vercel env vars are stale — push new keys to Vercel. If no errors at all and the SSE stream is empty: the response generator is yielding empty text-deltas, which usually means the model is refusing every input — check if `instruction.md` was just deployed with a typo.

**Follow-ups.** "How would you roll back?" (Vercel UI → Deployments → "Promote to Production" on the previous build.)

### Q5.3 — "Bedrock's Nova Micro is deprecated next year. What's your migration plan?"

**Model answer.** Three places to change. (1) `Config.model_id` default — one line. (2) Re-test `boto3-dataclass`'s typed cast against the new model's response shape. Different Bedrock model families return slightly different `output.message.content` structures. (3) Re-tune the system prompt — different models react differently to the same constraints, so the jailbreak defenses need re-validation. Then a small canary deploy: run 10% of traffic on the new model, compare response quality and cost via the (by-then-implemented) observability, full rollout if green.

**Follow-ups.** "Without observability today, how would you do that canary?"

### Q5.4 — "The chatbot starts answering questions outside of YingQiang's career. What's your incident response?"

**Model answer.** Three layers of defence to check. (1) Did the system prompt get truncated or modified? Re-deploy the known-good `instruction.md`. (2) Is the knowledge base being injected correctly as the first turn? Check `api/index.py` to confirm. (3) Has a new prompt-injection vector emerged that bypasses the existing jailbreak defenses? If yes, add the specific pattern to the system prompt (cheap mitigation) and consider regex-level input rejection as a backstop. Long-term: a small input-classifier (cheap LLM call) that gates whether the request even reaches Bedrock.

**Follow-ups.** "How would you know this is happening — what's the detection signal?"

### Q5.5 — "If you could only fix one thing about this project before showing it to a senior engineer, what would it be?"

**Model answer.** Tests. The whole project is well-architected and clean, but `tests_python/` and `tests_node/` are empty. A senior reviewing the repo will read the architecture and nod, then notice "no tests", and that single signal will dominate the impression. Even three or four tests covering `ai_sdk_adapter` snapshots and `ChatSession` history mutation would change the read substantially, because they'd prove the contracts are protected.

**Follow-ups.** "How long would those tests take to write?" "What's the next test after that?"
